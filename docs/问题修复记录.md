## 1. MOT20 验证脚本 (`validate.py`)

### 1.1 配置文件路径错误

**现象**：`FileNotFoundError: ... './ultralytics/cfg/trackers/yolojdetracker.yaml'`  
**原因**：使用相对路径 `./ultralytics/...`，工作目录非项目根时找不到文件。  
**处理**：在 `tracker/evaluation/mot_callback.py` 中基于 `__file__` 解析项目根目录，再 `os.path.join` 拼接 `ultralytics/cfg/trackers/{tracker_name}.yaml`。

### 1.2 进度条卡在 0%

**现象**：进度长时间停在 `0%|0/3`。  
**原因**：序列循环与逐图循环缺少可视进度。  
**处理**：对序列与单序列内图像分别使用 `tqdm`；对 `img_dir` 做 `.jpg`/`.png` 过滤；读图失败时 `continue` 并打印警告。

### 1.3 seqmap 缺失导致评估失败

**现象**：`TrackEvalException: no seqmap found: MOT20-train.txt`  
**原因**：未提供 seqmap，TrackEval 不知道评估哪些序列。  
**处理**：在 `dataset_root/seqmaps/` 下若不存在则自动生成 `MOT20-train.txt`：首行 `name`，后续每行一个序列名（与 `SPLIT_TO_EVAL: 'train'` 一致）。

### 1.4 模型预热（Warm-up）

首帧对每个序列做约 10 次空推理，用于拉起 CUDA/cuDNN/TRT 等初始化，避免把首次延迟算进正式测速；详见 `mot_callback.py` 中 `idx == 0` 分支。

### 1.5 修复后预期

- 任意工作目录下能加载 tracker 配置  
- 序列级 / 图像级进度可见  
- seqmap 自动就绪，HOTA、MOTA、IDF1 等可正常输出

## 2. Jetson / GPU 环境与依赖

### 2.1 Python 与 PyTorch 版本

JetPack 5（L4T R35）配套 GPU 版 PyTorch 多为 **Python 3.8** 的 NVIDIA 官方 wheel；若使用 **Python 3.10** 且仅从 PyPI 装 `torch`，常为 CPU 构建，在 Jetson 上无法用上预期 CUDA。

建议为 GPU 推理单独建环境（如 conda `yolo11-gpu`，Python 3.8），并从 NVIDIA 文档提供的 JP 版本 redist 安装 `torch`。

### 2.2 torchvision 与 torch 混装

使用 **NVIDIA `torch` wheel** 时，若直接 `pip install torchvision` 且不约束依赖，pip 可能 **覆盖** 为 PyPI 上的 CPU `torch`，或与当前 `torch` **ABI 不匹配**（后续 `torchvision.ops.nms` 等 C++ 扩展无法加载）。

**做法**：安装与 JP/torch 匹配的 `torchvision` 时可用 **`pip install torchvision ... --no-deps`** ，避免改写已装好的 `torch`。

## 3. ONNX 导出、推理后端与 IR 版本

### 3.1 ARM64 与 ONNX Runtime

PyPI 上通常 **没有** Jetson（linux aarch64）可用的 `onnxruntime-gpu`；`ultralytics/nn/autobackend.py` 中在 aarch64 + CUDA 环境下仅校验并依赖 **`onnx` + `onnxruntime`**，使用 **`CPUExecutionProvider`** 跑 ONNX，并打日志说明：若要 GPU 部署优先 **TensorRT engine** 或 **PyTorch**。

### 3.2 Jetson ONNX Runtime 的 IR 上限

部分设备上 ONNX Runtime 支持的 **IR 版本** 低于最新 `onnx` 导出默认值。`ultralytics/engine/exporter.py` 在保存前若检测到 **`model_onnx.ir_version > 11`**，将 **`ir_version` 降为 10**，避免加载失败。

## 4. 部署测速脚本 `benchmark_deploy.py`

- 请求 GPU 而环境无 CUDA 时做检查与提示，避免误导性测速。  
- 增加 **`free_cuda_memory()`**（如 `gc` + `torch.cuda.empty_cache()` + `synchronize`），在导出 TensorRT、切换前后端等阶段释放显存，降低 OOM 概率。  
- **TensorRT 建引擎**：`export_engine` 在 **`export_onnx` 后尽早 `del self.model`**，避免 PyTorch 与 TRT builder 同时占满显存。  
- **Jetson 根因与同进程必败**：父进程已加载 **PyTorch** 时，CUDA 显存“底座”常达 **5GB+**，TensorRT builder 在 tactic 阶段再申请 **~800MB** 即 `Device memory is insufficient` 并**段错误**；调小 workspace、改 tactic 来源、CPU 导出 ONNX **都不能**完全腾出这块已被 PyTorch 占住的上下文。  
- **Jetson 静态 FP32/FP16**（非 INT8、非 dynamic）：在 **`Exporter.__call__` 里、尚未执行 `deepcopy(model).cuda()` 时**即分支为 **`_export_jetson_engine_fully_isolated`**：① **子进程 1** 仅 `YOLO.export(format='onnx')`（进程退出释放该段显存）；② **子进程 2** 仅 `trt_isolated_builder.py` 建 engine。若仍在本进程先占满 GPU 再子进程建 TRT，**7.4GB 板子**上 TRT 单独即可能近满，必失败。环境变量 **`ULTRALYTICS_TRT_FULL_TAC=1`** 等仍作用于子进程 2。若单进程 TRT 仍 OOM，需 **降低 imgsz** 或在 **显存更大机器**上建 engine 再拷贝。  
- 导出 ONNX 时 **aarch64** 不拉取不存在的 **`onnxruntime-gpu`** pip 包（避免无意义重试）。

## 5. 后处理 NMS（Jetson + torchvision 不兼容）

### 5.1 现象

- `torchvision.ops.nms` 报错：无法加载自定义 C++ ops（与当前 `torch` 版本/构建不一致）。  
- 若回退实现不当：`benchmark_deploy.py` 走 ONNX（CPU ORT）+ 后处理时，**NMS 超 2s**，反复出现 `NMS time limit ... exceeded`。

### 5.2 原因归纳

1. **用 OpenCV `NMSBoxes` 时对数万候选做 `.tolist()`**：构造巨大 Python list，极慢。  
2. **仅用纯 PyTorch 贪心 NMS + 逐轮 `box_iou`**：在框大量互不重叠时，外层循环接近 **O(n)**，每轮再算 **O(k)** IoU，整体接近 **O(n²)**，在 Python 侧迭代数万次会长时间卡住。

### 5.3 处理（`ultralytics/utils/ops.py`）

- **`nms_xyxy`**：优先 **`torchvision.ops.nms`**。  
- 一旦出现 **`RuntimeError`（C++ ops 不可用）**，将模块级标志 **`_torchvision_nms_usable = False`**，后续帧**不再重复尝试** torchvision，减少异常开销。  
- 回退路径 **`_nms_xyxy_opencv`**：将 `xyxy` 转为 `xywh` **numpy**，**直接**把 `xywh` 与 `scores` 的 **ndarray** 传给 **`cv2.dnn.NMSBoxes`**（**不要**对整批框 `.tolist()`），再把保留的下标 **映射回** 原 `boxes` 的 device。  
- 实测：数万候选下该路径可在毫秒～数百毫秒量级完成，满足 `non_max_suppression` 内时间上限。

### 5.4 `task="jde"` 与 ONNX / Engine（必读）

加载 **`.onnx` / `.engine`** 时若**未**指定 `task="jde"`，会走 **detect** 后处理：`non_max_suppression` 得不到正确 **`nc`**，把 **ReID 嵌入通道**与分类混在一起求 `amax`，候选框数量暴涨，NMS（含 OpenCV 路径）耗时极长并触发 **NMS time limit**。

**处理**：脚本与业务代码中统一 `YOLO(path, task="jde")`；本仓库 **`scripts/benchmark_deploy.py`** 已对齐。**JDEPredictor** 对 `AutoBackend` 单张量输出须 `preds` 整tensor 入 NMS，见 `ultralytics/models/yolo/jde/predict.py`（`list/tuple` 才取 `[0]`）。

仍可消除 “assuming task=detect” 告警；与 §5.3 NMS 回退策略相互独立。

## 8. Git 推送与 GitHub 鉴权

在 Jetson / 远程开发环境向 GitHub 推送时，常见两类错误：**HTTPS 凭据失败**、**SSH `Permission denied (publickey)`**。以下归纳原因与可行处理（与项目代码无关，属本机 Git / 网络环境配置）。

### 8.1 HTTPS：`Missing or invalid credentials` / `ECONNREFUSED ... vscode-git-*.sock`

**现象**：`git push` 报缺少凭据；或连接 `/run/user/.../vscode-git-....sock` 时出现 `ECONNREFUSED`；远端提示 `No anonymous write access`、鉴权失败。

**原因**：

1. GitHub **不允许匿名写**远程仓库；HTTPS 必须提供有效身份（**Personal Access Token**，或已正确保存的凭据），不能再用账户登录密码推送。
2. Git 可能配置为通过 **Cursor / VS Code 的凭据助手**（Unix 套接字）获取令牌；当前 shell 环境中 **没有进程监听该 socket**（例如使用外部 SSH 终端、会话不一致）时，会 `ECONNREFUSED`，进而表现为无可用凭据。

**处理（任选或组合）**：

| 做法 | 说明 |
|------|------|
| 改用 SSH | `git remote set-url origin git@github.com:用户名/仓库.git`，使用 SSH 密钥，不依赖 VS Code 的 HTTPS 助手。 |
| 继续用 HTTPS | 在 GitHub 创建含 `repo` 权限的 **PAT**，推送时在密码处粘贴令牌；若凭据助手异常，可对当前仓库 `git config --local credential.helper store`（或按需清空有问题的 helper）后再推送。 |
| 在 Cursor 集成终端重试 | 有时可正常唤起凭据；若仍失败，以 PAT 或 SSH 为准。 |

### 8.2 SSH：`Permission denied (publickey)`

**现象**：`git@github.com: Permission denied (publickey).`，无法读取远程仓库。

**原因**：

1. 私钥文件名 **不是** SSH 默认会尝试的名字（如自定义为 `id_ed25519_github`），未配置时 OpenSSH **不会自动选用**该文件。
2. 未在 **`~/.ssh/config`** 中为 `Host github.com` 指定 `IdentityFile`。
3. 若密钥带口令，**`ssh-agent` 未运行**或未 `ssh-add` 对应私钥，也会导致认证失败。

**处理**：

1. 将 **公钥** 添加到 GitHub：**Settings → SSH and GPG keys**。
2. 编辑 **`~/.ssh/config`**（权限建议 `chmod 600`），例如：

```sshconfig
Host github.com
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_ed25519_github
  IdentitiesOnly yes
```

将 `IdentityFile` 换成你实际的私钥路径；`IdentitiesOnly yes` 可避免与多把密钥混用时选错钥匙。

3. 需要代理加载带口令的密钥时：

```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519_github
```

4. 自检：`ssh -T git@github.com`，出现 “successfully authenticated” 即表示 SSH 侧已通过（部分环境下该命令退出码仍为非 0，可忽略）。

### 8.3 小结

| 错误类型 | 根因 |
|----------|------|
| HTTPS 鉴权失败 | 未向 GitHub 提供有效令牌，和/或 VS Code 凭据 socket 不可用 |
| SSH `publickey` 失败 | 未让 ssh 使用正确的私钥（配置、默认文件名、agent） |

## 总结

- **验证链路**：路径、进度、seqmap、预热等问题已在 `mot_callback.py` 侧闭环。  
- **Jetson 部署**：注意 **Python/torch/torchvision** 组合与 **ONNX 为 CPU EP** 的现实；GPU 推理优先 **PyTorch 或 TensorRT**。  
- **NMS**：在 **torchvision C++ 不可用** 时，用 **OpenCV + numpy 直通**，避免 `.tolist()` 与纯 Python 双重循环的 **O(n²) 最坏情况**。  

以上修改在功能上尽量保持兼容，便于在桌面与 Jetson 上共用同一套仓库。

**Git / GitHub**：推送失败时先区分 HTTPS 与 SSH，再按 §8 检查凭据与 `~/.ssh/config`。
